PCS Online Catalog - Functional and Technical Documentation

Repository

https://github.com/thomastorvpcs/onlinecatalog

1. Purpose and Scope

PCS Online Catalog is a web application used to browse live device inventory, filter by multiple criteria, build request carts, and submit inventory requests. It supports buyer and admin workflows and integrates with external inventory APIs for synchronization.

2. Functional Documentation

2.1 User Roles

  • Buyer: sign in, browse catalog, search and filter devices, open device details, add requested items, and submit requests.
  • Admin: all buyer capabilities plus user administration, approvals, catalog utilities, weekly special banner controls, and external inventory sync operations.

2.2 Core User Flows

  • Authentication: login, create account, forgot password with verification code, and approval-required account state.
  • Catalog browsing: home page by category, category detail pages, dynamic filters, and server-side pagination.
  • Device details: modal view with image carousel, availability by location, compact specs, and request creation controls.
  • Requested items: slide-in request builder with editable price/quantity, per-line totals, highlighted grand total, and submit request action.
  • Request submission and estimate simulation: request is persisted in backend and a dummy NetSuite estimate is created for workflow validation.
  • Admin operations: user create/delete/approve, admin role toggles, catalog clear/seed endpoints, weekly special controls, and inventory sync trigger.
  • Admin request status controls: admin can set dummy estimate/request status (New, Received, Estimate Created, Completed) from the request details panel.

2.4 Request Modal Fulfillment UX (Current)

  • Order location selector: user must select a source location before submitting a request.
  • Full-fulfillment default: location dropdown initially shows only locations that can fully fulfill all requested lines.
  • Partial-location toggle: user can enable an option to include locations that only partially fulfill the request.
  • Partial-location labels: locations that cannot fully fulfill are marked in dropdown as (Partial).
  • Line-level shortage guidance: for selected partial locations, affected lines show shortage details directly under the product name.
  • Quick adjustment action: each shortage message includes an action to set quantity to max available for that location.
  • Submission protection: request submission is blocked until all shortage lines are resolved for the selected location.
  • Quantity edit behavior: quantity input supports clearing/free typing, then normalizes to valid range on blur.
  • Currency formatting: request modal totals and request list/detail monetary values are formatted with thousands separators.

2.3 Business Rules (Current)

  • Only approved users can fully access inventory APIs; unapproved users are redirected to a waiting-for-approval experience.
  • Admin role is required for user management and catalog/integration administrative actions.
  • Startup seeding can auto-populate realistic catalog rows for deploy/test readiness.
  • Category display order is fixed: Smartphones, Tablets, Laptops, Wearables, Accessories.

3. Technical Documentation

Frontend

  • Main UI and state logic: src/App.jsx
  • Frontend entrypoint: src/main.jsx

Database

  • Schema: backend/db/schema.sql
  • Seed data: backend/db/seed.sql

Documentation

  • OpenAPI spec: docs/openapi.yaml
  • Inventory API notes: docs/inventory-api.md
  • Project overview: README.md

3.1 Auth Model

  • Bearer access token for API calls
  • Refresh-token rotation for session continuation

Implementation reference: backend/server.mjs

3.2 Roles

  • buyer

3.3 Admin-only Capabilities

4. API Surface

4.1 API Groups

  • /api/auth/* - login/register/me/refresh/logout/reset
  • /api/devices - search/filter/pagination
  • /api/requests - create/list request quotes
  • /api/inventory/* - inventory updates/events
  • /api/users - admin user management
  • /api/admin/* - admin controls
  • /api/integrations/boomi/inventory/sync - external inventory pull
  • /api/integrations/netsuite/estimates/dummy - dummy estimate creation
  • /api/integrations/netsuite/estimates/dummy/status - dummy estimate status updates

Full schema and examples: docs/openapi.yaml

4.2 Device Query Capabilities

  • search

  • modelFamily
  • grade
  • region
  • storage

6. External Inventory Integration

6.1 Preferred Integration (Current)

OAuth2 Client Credentials + subscription key

Required env vars:

  • INVENTORY_API_URL
  • INVENTORY_SUBSCRIPTION_KEY
  • INVENTORY_OAUTH_TOKEN_URL
  • INVENTORY_OAUTH_CLIENT_ID
  • INVENTORY_OAUTH_CLIENT_SECRET
  • INVENTORY_OAUTH_SCOPE

Flow:

  1. Backend fetches OAuth access token.
  2. Backend calls inventory endpoint with bearer token + subscription key.
  3. Response is normalized and upserted into local catalog/inventory tables.

Implementation reference:

  • backend/server.mjs
  • docs/inventory-api.md

6.2 Legacy Fallback

If OAuth config is absent, legacy Boomi basic-auth fallback can be used:

  • BOOMI_INVENTORY_URL
  • BOOMI_CUSTOMER_ID
  • BOOMI_BASIC_USERNAME
  • BOOMI_BASIC_PASSWORD
  • BOOMI_EXTRA_AUTH (optional)
  • BOOMI_TLS_INSECURE (local test only)

7. Admin Catalog Operations

Available admin endpoints:

  • POST /api/admin/catalog/clear
  • POST /api/admin/catalog/seed-test
  • POST /api/admin/catalog/seed-real
  • POST /api/admin/catalog/apply-image-mapping

7.1 Seed Modes

  • seed-test: synthetic bulk dataset
  • seed-real: realistic model-family dataset with storage/color/grade variants and product-image pools

Implementation: backend/server.mjs

8. Frontend Technical Behavior

  • SPA state-driven routing in App.jsx (no React Router currently)
  • Modal overlays for:
    • device details
    • requested-items side sheet
    • session-expiry warning
  • Dynamic filters and server-side category pagination
  • Skeleton loading states for async fetches
  • Image fallback component to avoid broken product thumbnails
  • Request cart modal includes fulfillment-aware location selector and per-line shortage reconciliation actions.
  • Grand total and inline confirmation messages are intentionally emphasized for request feedback visibility.

Implementation:

  • src/App.jsx
  • styles.css

9. Security and Operations

  • Passwords are never stored in plaintext.
  • Role checks enforced server-side.
  • Secrets must be injected via environment variables.
  • Local-only insecure TLS toggle supported for test cert chains.

Recommended:

  • Rotate exposed credentials immediately if shared in logs/chat.
  • Keep production with strict TLS validation.

10. Deployment Runbook (Render)

  1. Push changes to GitHub (main).
  2. Ensure Render service points to correct repo/branch.
  3. Configure required INVENTORY_* environment variables.
  4. Configure deploy startup seed behavior (optional):
    • AUTO_SEED_REAL_ON_STARTUP (default true)
    • DEPLOY_REAL_SEED_COUNT (default 100, max 1000)
  5. Redeploy/restart service.

Validate:

  • login
  • /api/auth/me
  • /api/devices
  • /api/integrations/boomi/inventory/sync (admin)

Related config/docs:

  • README.md
  • docs/openapi.yaml

11. Troubleshooting

401 Unauthorized on API calls

Access token missing/expired; refresh or re-login.

500 Inventory credentials are not configured

Missing required INVENTORY_* vars in running backend process.

Catalog appears empty after deploy

Verify startup seed settings:

  • AUTO_SEED_REAL_ON_STARTUP=true
  • DEPLOY_REAL_SEED_COUNT set to desired value (e.g. 100)

self-signed certificate in certificate chain

Upstream TLS chain issue in local testing; use local-only TLS bypass setting where appropriate.

Frontend MIME/module errors

Ensure app is run with Vite (npm run dev) or built/previewed correctly; avoid static serving raw JSX entrypoints.
